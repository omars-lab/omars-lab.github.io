How the docs sidebar distinguishes **draft** docs from **published** ones while
authoring locally, without ever leaking that distinction (or the drafts
themselves) to production.

<!-- truncate -->

## Problem

Almost half of this site's docs (~140 of ~289) carry `draft: true`. Docusaurus
treats drafts asymmetrically:

- **Production build:** draft docs are excluded entirely, with no route, not in any
  sidebar, not in autogenerated index pages.
- **Dev server (`yarn start`):** drafts **are** rendered and **appear in the
  sidebar**, indistinguishable from published docs.

So while authoring, the "Explore" sidebar is a mix of shipped and in-progress
content with no visual cue which is which. The goal: badge drafts in the sidebar
**on localhost/dev only**, so the dev experience is informative and production is
untouched.

## Background: what Docusaurus gives us

`draft: true` is the upstream mechanism for "work on this in the repo without
exposing it in prod" ([Docusaurus: Create a doc](https://docusaurus.io/docs/create-doc),
[Docusaurus PR #6457](https://github.com/facebook/docusaurus/pull/6457) that added
draft exclusion from build + sidebars). It is the **single source of truth** we
build on. Confirmed behavior: drafts stay visible + navigable in dev, vanish from
prod ([Docusaurus issue #3417](https://github.com/facebook/docusaurus/issues/3417)).

Two ways to get per-item draft state to a swizzled sidebar item:

1. **`sidebar_custom_props`**: frontmatter custom metadata that Docusaurus
   propagates to the client and exposes on the sidebar item
   ([Docusaurus: Sidebar items](https://docusaurus.io/docs/sidebar/items)). A swizzled `DocSidebarItem/Link` could
   read `item.customProps.draft`. **But** it requires adding a *second* field
   (`sidebar_custom_props: {draft: true}`) to every draft doc and keeping it in
   sync with `draft: true`, duplicate state that will drift.
2. **A small content plugin** that reads each doc's existing `draft: true`
   frontmatter at load time and publishes the set of draft permalinks as global
   data. One source of truth (`draft:`), nothing to keep in sync.

**Decision: the plugin.** It avoids duplicate frontmatter and derives everything
from the field authors already set. The `sidebar_custom_props` route is only
preferable if we later want per-doc badge *variants* (e.g. "draft" vs "wip" vs
"review") that aren't expressible from a boolean, noted as a future option.

## Approach

Dev-only, two pieces, both hard-gated so nothing reaches production:

### 1. `plugins/draft-docs`: publish draft permalinks

A local Docusaurus plugin walks `docs/`, reads frontmatter (`gray-matter`), and for
every `draft: true` file computes its permalink, exposing the list via
`setGlobalData`. Permalink derivation mirrors Docusaurus for the docs plugin
(default `routeBasePath: 'docs'`):

- Build from the file's **directory path** with the conventional `NN-` ordering
  prefixes stripped (`2-definitions` → `definitions`).
- A `slug:` renames the **last** segment only (it does not replace the whole path):
  `2-definitions/README.md` + `slug: definitions` → `/docs/definitions/definitions`.
- `README`/`index` files resolve to their directory.

**Validation:** the computed permalinks were diffed against Docusaurus's own
`.docusaurus/globalData.json` doc paths: **139/140 match**. The single non-match is
the experiments `_TEMPLATE` placeholder (`experiment-<flag-key>`), which has no real
route, so a stray badge is impossible. (Docusaurus's client `draftIds` is empty in
dev and the client doc data carries no `frontMatter`, so reading draft state from
global data alone isn't possible in dev, hence the plugin.)

### 2. Swizzled `@theme/DocSidebarItem/Link`: render the badge

- **`draftBadge` helper**: `useIsDraft(href)` reads the plugin's global data and
  returns `true` only when **all** of: `process.env.NODE_ENV !== 'production'`
  (build-time, tree-shakeable), `isLocalhost()` (runtime, reused from
  `src/experiments.ts`), and the normalized `href` is in the draft set.
- **`Link` (leaf docs only)**: a minimal fork of the upstream component that appends
  a `<DraftBadge/>` (a compact "D" pill; full word in `title`/`aria-label`, so it
  never wraps next to long labels) after the label.

### Leaf docs only: why categories aren't badged

An earlier version also badged categories, but it was both buggy and misleading:

- A folder whose **index** is a draft can still appear in prod if it has any
  **published child** (e.g. `Definitions`, whose index 404s in prod, but the folder
  shows because one child is published). Badging the folder "draft" implies the whole
  folder is gone, which is false.
- A **fully**-draft folder (every descendant draft → folder truly absent from prod)
  usually has **no index page**, so its sidebar entry's `href` is `undefined`
  (confirmed by logging `item.href` from the swizzle: index-less categories →
  `undefined`). With no href there's nothing to match on, so it can't be badged
  without fragile label/emoji matching.

So categories are left unbadged; only leaf draft docs get the "D". Leaf badging is
reliable (every draft doc has a stable permalink) and is what actually answers
"which entries won't ship."

### Why dev-only is safe (defense in depth)

1. Drafts don't exist in the prod build at all (Docusaurus excludes them).
2. The plugin's global data is therefore empty of any rendered route in prod.
3. `useIsDraft` short-circuits on `NODE_ENV === 'production'` (tree-shaken) and on
   non-localhost hosts.

## Testing

- **`test/e2e/draft-sidebar.spec.ts`** (dev project, :3000): a draft leaf doc shows
  the "D" badge, the badge sits on a leaf link (no category markers), and badging is
  selective (a published sibling has none).
- **`test/e2e/dev-only-surfaces.spec.ts`** (prod project, :4173 build): asserts the
  **absence** in production: no draft badges (`span[aria-label="draft"]`), and draft
  routes 404. This is the localhost-only guarantee, enforced (the same spec also
  guards the DebugMenu FAB).

## Future / open

- **Naming convention + enforcement.** With ~48% of docs in draft, a lightweight
  convention (and possibly a Claude Code hook that checks frontmatter consistency
  when `draft: true` is present) is worth considering, tracked separately.
- **Badge variants** (`draft`/`wip`/`review`) would justify switching to
  `sidebar_custom_props`.
- **Author dashboard**: the same draft-permalink data could power a "drafts to
  finish" list (overlaps the existing `publish-site` triage skill).
