# Techniques

This is the durable index of the **practical techniques** I use to build software: the
self-contained how-to methods for doing one thing well. Each entry answers a "how do I do X?"
question, from wiring a CI/CD pipeline to managing secrets safely.

Unlike a project log, a technique is not a dated thing I did; it is a return-to method that
stays useful across projects. So it lives here in Craft, not on the Initiatives blog. This
page groups the techniques by area so you can scan the whole set at once. Move a technique
between areas by editing its `area` frontmatter, and it re-sorts here automatically.

## What counts as a technique here

A technique is a practical method for the software development lifecycle:

- 🔨 **Build and deployment** methods for shipping applications.
- 🧪 **Testing and quality** approaches for ensuring code holds up.
- ⚙️ **CI/CD** practices for continuous integration and deployment.
- 🔒 **Security** practices, including managing secrets and passwords safely.
- 🔄 **Workflow** improvements that streamline how the work gets done.
- 📦 **Dependency management** for keeping libraries and versions in order.

## How this differs from the neighbours

These are all "software development" methods, but they split by intent:

- **Techniques** (here) focus on building, testing, and deploying software: the lifecycle and
  code delivery.
- **[Patterns](/craft/software-development/patterns)** are about how tools and pieces **compose**
  into a working whole (a build pipeline, a documentation stack).
- **[Scripting](/craft/software-development/scripting)** is command-line tools and automation scripts.
- **Documentation and blogging** methods (creating and publishing content) live under
  [Blogging](/craft/blogging).
