# Plan: Design-System reference pages under `/handbook/design-system/*`

## Context

The "Bytes of Purpose Design System" lives as a claude.ai/design project
(`a05a9fd6-…`) whose `guidelines/*.html` cards specimen the brand's foundations
(Colors, Type, Spacing, Brand/Logos). None of that is reflected **on the site**.
The ask: publish those foundations as first-class Handbook pages at
`https://blog.bytesofpurpose.com/handbook/design-system/{Colors,Typography,Spacing,Logos…}`,
mirroring how the design project groups them.

**Decision (confirmed with user):** render specimens as **native React components**
that read the repo's OWN live tokens from `src/css/custom.css` (theme-aware,
drift-proof, passes the DS-token + contrast gates), NOT embedded claude.ai HTML.
**Pages to build (confirmed):** Colors, Typography, Spacing & Elevation, Logos & Iconography.

The design project's `readme.md` is explicit that `src/css/custom.css` is the
**production source of truth** and the DS files are a reverse-engineered mirror
(with a "Reconciliation notes" section flagging where the DS overstates what ships,
e.g. hairline renders `#ebedf0` not `#dde2df`). So these pages document the *repo's*
tokens, honoring those repo-reality caveats.

This is a new `/handbook` docs sub-tree. The handbook sidebar is
`{type:'autogenerated', dirName:'.'}` (`sidebars-handbook.js`), so a new
`docs/handbook/design-system/` folder with a `_category_.json` appears automatically.

## Approach

### 1. Specimen components — new `src/components/DesignSystem/`

A small, self-contained kit (CSS-module + theme tokens, no new colors invented):

- `ColorSwatch.tsx` — one chip: live `background: var(--token)` fill + name + hex +
  token label + optional note. Reads the *live* CSS var so light/dark are automatic.
  A `<ColorRow>` wrapper lays chips in a responsive row/grid.
- `TypeSpecimen.tsx` — renders a sample line in a given family/size/weight/tracking,
  with the token name + resolved value shown as caption. A `<TypeScaleTable>` variant
  walks `--text-*`.
- `SpacingScale.tsx` — bars sized by `var(--space-N)` with the token + rem value.
- `RadiusElevationDemo.tsx` — swatch tiles showing each `--radius-*` and the two-step
  shadow (`--shadow-sm` rest, `--shadow-md` hover) + `--lift-*` on hover (reduced-motion
  guarded per discipline rule 6).
- `TokenTable.tsx` — a shared small primitive for the "token · value · purpose" tables
  (keeps the four pages consistent).
- `index.ts` re-exports; `styles.module.css` holds the layout (uses `--space-*`,
  `--radius-md`, `--border-hairline`, `--surface-card` — all existing tokens).

All values come from tokens; the only literal hexes are the *labels* (the human-readable
"#3C7256" caption text), matching how the source `guidelines/*.html` cards read.

**Reduced-motion:** the hover-lift demo self-guards with
`@media (prefers-reduced-motion: reduce)` in its module (discipline rule 6). No auto motion.

### 2. Register in MDX — `src/theme/MDXComponents.tsx`

Add the specimen components to the global MDX map (same pattern as `Card`, `UsedIn`,
`KanbanBoard` already there) so the `.mdx` pages can use them without per-file imports.

### 3. The four pages + landing — `docs/handbook/design-system/`

- `_category_.json` → `{ "label": "🎨 Design System", "position": 4 }` (after
  🎛️ Components at position 3; Terminology is 3 today but positions can repeat across
  category vs docs — will set 4 to sit cleanly last among the reference groups).
- `README.mdx` → `slug: /design-system`, `kind: showcase`, a short landing that states
  the source-of-truth model (custom.css) + links the four sub-pages. Mirrors
  `components/README.mdx` structure.
- `colors.mdx` → `slug: /design-system/colors` — Brand green scale, Tea-party pastels
  (with the **accent-fill-only** discipline callout), Premium gold, Surfaces & ink.
  Uses `<ColorRow>`/`<ColorSwatch>`. Documents the `#ebedf0` hairline repo-reality note.
- `typography.mdx` → `slug: /design-system/typography` — Fraunces display, Geist body,
  Geist Mono + eyebrow; the `--text-*` scale, weights, tracking, line-heights via
  `<TypeSpecimen>`/`<TypeScaleTable>`.
- `spacing.mdx` → `slug: /design-system/spacing` — `--space-1..9` via `<SpacingScale>`,
  `--radius-*` + the quiet two-step elevation + `--lift-*`/motion tokens via
  `<RadiusElevationDemo>`.
- `logos.mdx` → `slug: /design-system/logos` — the lightbulb `</>` and binary-pyramid
  marks (`static/img/logo.svg`, `logo-binary.svg` + dark variants), the feature icons
  (`static/img/{engine,artifacts,posts,support}.svg`), the glyph/emoji conventions
  (→ ★ ☕ ·), and the cathedral-arch illustration set (`static/img/cards/*.png`).

**Frontmatter contract per page** (verified against existing showcases): absolute
`slug:`, `title:` with a leading emoji, `description:` (50–160 ch for SEO), `kind: showcase`,
`draft: false`, short `sidebar_label`. **No em-dash** in any reader-facing copy (use `·`,
commas, parens) — the `em-dash-voice-hook.sh` blocks it.

## Conventions honored (from CLAUDE.md)

- **Reach for a token, not a literal** — every specimen fill/size/space is `var(--token)`;
  `make validate-ds-tokens` should stay clean. Ran the `implement-with-design-system` skill (Step 0
  DS pull done: `custom.css` matches the DS `tokens/*`).
- **Every interactive component gets a visual + mobile pass** — will check the four pages +
  specimen components at 375px and desktop before the commit (tap targets, no h-overflow,
  swatches reflow, ≥16px body).
- **Track work as tasks** — one task per page/component group.
- **commit → PR → ask to merge → squash** — single feature branch
  `feat/handbook-design-system`, one PR, verify locally, ask before merge.
- Structure-validator: new docs carry absolute slugs + descriptions; run
  `make validate-structure` + `make validate-seo` + `make check-contrast` + `make validate-ds-tokens`.
