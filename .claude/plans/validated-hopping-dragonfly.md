# Import personalbook roles as consolidated `/journey/roles/` posts (read-only, new skill)

## Context

**The problem.** A role-import mechanism already exists (personalbook's `scan-blog-worthy` skill),
but it lives in personalbook and **writes markers back into your private role files** (`The Analyzer/
Activities.md` now carries `blog: yes` / `blog_slug:` frontmatter + a "Published as a blog draft"
note), points at a **stale/wrong blog path**, and produces **flat `/blog/` posts** instead of durable,
consolidated role posts. A role is a rich, lasting thing — its outline, the skills you'd use, the
artifacts you'd produce, the habits you'd keep, why it matters — and that belongs consolidated in one
durable post under `/journey/roles/`.

**The outcome (revised).** Leave `scan-blog-worthy` **entirely alone** — no move, no refactor. Build
a **new skill in THIS repo**, `import-personalbook-role`, that **reads** the personalbook role
material (`Activities.md` files *and* the structured output: `Overview.md`, `skills/`, `artifacts/`,
`habits/`, `knowledge/`) purely as **influence/source** for a consolidated `docs/journey/roles/<role>.md`
post it authors here. Strictly **read-only on personalbook** — it never writes a marker back. Add the
**hooks + validations** this repo expects so the process is fail-closed (private content can't leak,
role docs are well-formed, redirects/structure stay valid).

## Why a role belongs in `/journey/roles/` (durable docs)

Per the repo's durable-vs-temporal tenet, a *role* is **durable** (how I habitually operate / who I am
when I act). So each role is a **docs** page in the `journey` instance, not a dated `/initiatives`
post. The journey sidebar (`sidebars-self.js`) is **autogenerated** from `docs/journey/`, so adding a
`roles/` topic folder needs **no sidebar edit** — just the folder, a `_category_.json`, a README
landing, and one doc per role.

The existing narrow post `a-framework-for-quantifying-yourself` (a genuine standalone *framework*)
**stays as-is** in `/initiatives`; the new `/journey/roles/the-analyzer` doc is the role-level
consolidation and links out to it.

## What "a role" consolidates into (the role-doc shape)

The personalbook role folder is the **source/influence**, read but never mutated. The
`/journey/roles/<role>.md` doc mirrors its anatomy, reader-facing and privacy-hardened:

- **Why this role matters** — from `Overview.md` `## Core Philosophy` (the role's reason-for-being).
- **What the role does** — Key Focus Areas / essence.
- **Skills I use in this role** — from `skills/*/SKILL.md` (name + essence; transferable process only).
- **Artifacts I produce** — from the **public-safe** `artifacts/` (the *kinds* of things I make);
  **never** the private intent triad `obligations.md` / `desires.md` / `motivations.md`, nor
  family/receipt/dated-personal application files.
- **Habits I keep** — from `habits/*/HABIT.md` (cadence + what it exercises).
- **What I know / mental models** — from `knowledge/` where transferable.
- **Role evolution** (optional) — aliases/history from `Overview.md`.
- Links out to any spun-off `/initiatives` framework posts for the role.

`Activities.md` is read as **additional influence** (the raw activity/planning-question material) —
it enriches the "what the role does" and can seed a spun-off framework post, but is not copied wholesale.

## Approach

### 1. New read-only skill: `import-personalbook-role`

`.claude/skills/import-personalbook-role/SKILL.md`, authored fresh (NOT moved from personalbook). It:

- **Reads** a role folder from personalbook (`Overview.md` + `skills/`/`artifacts/`/`habits/`/
  `knowledge/` + `Activities.md`) as influence. **Never writes to personalbook** — this is the
  load-bearing contract, stated up front and enforced (see §3 hook).
- **Consolidates** into `docs/journey/roles/<role-kebab>.md` following this repo's `author-blog-post`
  frontmatter conventions (instance-relative `slug: /roles/<role-kebab>`, `authors: [oeid]`, `tags:`,
  `description:` ~50–160ch, `draft: true` to start) and the **no-em-dash** content rule.
- **Excludes private artifacts by default** — reuses the leak-signal instinct (`obligations`/
  `desires`/`motivations`/family/finance/medical) so nothing private lands in a public role doc.
  Records what it excluded (auditable trail).
- **Self-audits against personalbook's role-structuring skills** so it never drifts from how a role is
  actually shaped: a **Step 0 reconcile** reads the `SKILL.md` descriptions of `extract-role-skills`,
  `establish-role-habits`, `establish-role-learning`, `structure-role-purpose`,
  `extract-obligations-desires` (they define what each role subfolder *is* and which files are the
  private triad to exclude), caching a small **`role-anatomy.json`** keyed to personalbook's HEAD.
  Re-reconcile only when those skills change (git-commit-keyed, mirroring how this repo caches).
- Keeps a blog-side **`import-ledger.json`** (one entry per role: `source_role_path`, `journey_doc`,
  `spun_off_posts`, `status`, `excluded_files`, `reconciled_at_personalbook_commit`). Committed here;
  personalbook stays untouched.
- Hands off to the existing content skills: `author-blog-post` (frontmatter/MDX), `upgrade-post`
  (weave components), `link-glossary-terms`, `name-post` (title voice).

### 2. The `/journey/roles/` docs topic

- `docs/journey/roles/_category_.json` → `{ "label": "🎭 Roles", "position": <unused> }`.
- `docs/journey/roles/README.mdx` → landing (`slug: /roles`, pinned `sidebar_label: '👋 Welcome'`,
  `sidebar_position: 0`), explaining the roles model + indexing the role docs.
- `docs/journey/roles/the-analyzer.md` → the **worked example** (import The Analyzer end-to-end).

### 3. Hooks + validations this process needs (the fail-closed guards)

Following this repo's established pattern — a `scripts/validate-*.js` run two ways: a **warn-tier
PostToolUse `Write|Edit` hook** (`--file`-scoped, exit 0, registered in `.claude/settings.json`) plus
a **blocking `make validate-*` gate** (exit 2). Guards to add:

| Guard | Kind | What it enforces |
|---|---|---|
| **Role-leak guard** (`validate-role-privacy.js` + hook) | **BLOCKING** (this is the secure-by-default one) | A doc under `docs/journey/roles/` must contain **no private-artifact content** — scan for the private-triad signals (obligations/desires/motivations phrasing, family names incl. Yara, finance/medical/employer markers). A hit **exits 2**. Mirrors the repo's fail-closed tenet: a role post can't ship private residue by omission. |
| **Role-doc structure** (`validate-role-doc.js` + warn hook) | warn-tier | A `journey/roles/*` doc has the required sections (Why this role matters / Skills / Artifacts / Habits) + an absolute instance-relative slug + a `description:`. Missing section → warn; the blocking gate is `make validate-roles`. |
| **Read-only-personalbook guard** (`personalbook-read-only-hook.sh`) | **BLOCKING** | A PreToolUse `Write|Edit` hook that **blocks (exit 2)** any write whose path is under `…/git/personalbook/` while this skill is the active context. Makes the read-only contract impossible to violate by omission (same shape as personalbook's own `noteplan-no-drop-hook.sh` fail-closed model). |
| **Reuse existing gates** | — | The role docs automatically fall under `em-dash-voice-hook.sh` (blocking on `—`), `validate-docs-structure-hook.sh` (absolute-slug), `validate-seo-hook.sh` (description health), `validate-redirects` (if a role doc is ever moved), and `validate-draft-hook.sh`. No new work — just confirm they cover `docs/journey/roles/`. |

Each new validator is added to the `Makefile` (a `validate-roles` / `validate-role-privacy` target,
folded into the aggregate validate + the deploy pre-flight), its hook registered in
`.claude/settings.json`, and — per repo convention — the CLAUDE.md operating-convention note updated
in the SAME change if it introduces a durable rule (the role-leak guard is a secure-by-default
addition worth a short tenet cross-link).

### 4. Worked example: import The Analyzer

Import **The Analyzer** (already fully structured) → `docs/journey/roles/the-analyzer.md`,
consolidating its 5 skills, public artifacts, 2 habits, knowledge, and Core Philosophy, linking out to
the existing `a-framework-for-quantifying-yourself`. Prove the leak guard bites (a planted private
line → `make validate-role-privacy` exits non-zero), then remove the plant.

### 5. Index + memory

- Add the skill to the **Skills map** table in `CLAUDE.md` (import personalbook role → `/journey/roles/`,
  read-only, self-audits against personalbook role-structuring skills; guarded by the new validators).
- Add a short **operating-convention / tenet note** in `CLAUDE.md` for the role-leak guard (fail-closed).
- Add a memory pointer: the personalbook↔blog boundary is **read-only** (no write-back), roles are
  durable `/journey/roles/` docs, the self-audit anti-drift step, and the new guards.

### 6. Personalbook: no changes in this workstream

`scan-blog-worthy` and the role-structuring skills are **untouched** (no move, no refactor, no
deprecation edit). Separately, the `blog:` markers the old flow already wrote into personalbook role
files remain **your call** — I can propose a one-time read-then-strip cleanup as a *follow-up* if you
want, but it's out of scope for this plan unless you ask.

## Verification

1. **Personalbook is unmutated:** after importing The Analyzer, `git -C personalbook status` shows
   **no changes**; the read-only hook blocks any attempted write under `…/personalbook/`.
2. **Role doc builds + renders:** `make start`, visit `/journey/roles/the-analyzer` and `/journey/roles`
   — renders, appears in the autogenerated journey sidebar under 🎭 Roles, no build error.
3. **Leak guard bites (fail-closed proof):** plant a private line in the role doc → `make
   validate-role-privacy` **exits non-zero** and the PreToolUse/PostToolUse hook flags it; remove the
   plant → passes clean. The ledger's `excluded_files` lists the skipped private artifacts.
4. **Structure guard:** `make validate-roles` passes on the finished doc; dropping a required section
   warns.
5. **Existing gates pass:** `make validate-structure`, `make validate-seo`, `make validate-redirects`,
   and a `make build` (em-dash hook must not fire).
6. **Anti-drift works:** the Step 0 reconcile produced `role-anatomy.json` keyed to personalbook HEAD;
   forcing a re-reconcile re-reads the structuring skills' definitions.
