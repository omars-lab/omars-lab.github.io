# Manage the changelog

The **Changelog** is the durable, append-only record of completed work, the thing the live task
list is NOT. Completed tasks are ephemeral (they get deleted to keep the list short); the
changelog is where their substance lives forever. It renders as a **timeline + heatmap + filter**
UI at **`/changelog`** (its own blog instance), reachable from the **📓 Changelog** link in the
**Handbook** sidebar and from the footer.

This skill owns the changelog domain: the file format, the generator, the build-safety rules for
the summaries (they render on-site), and the archive loop. The TRIGGER and the high-level loop
also live in the CLAUDE.md "archive completed tasks to the changelog" convention; this skill is
the detailed how-to that convention points at. Don't duplicate, cross-reference.

## The pieces

| Piece | What it is |
|---|---|
| `bytesofpurpose-blog/changelog/CLAUDE-CHANGELOG.md` | the DURABLE source: one `## ` batch per archive event (append-only). Its header documents the format. |
| `scripts/generate-changelog-data.js` | splits CLAUDE-CHANGELOG.md on the `## ` batch headings → one card per batch → `src/components/Changelog/changelog-data.json` (gitignored, generated; never hand-edit). Also folds in the per-file changelog posts under `changelog/content/*` + `changelog/development/*`. |
| `src/components/Changelog/*` | the `<Changelog>` timeline/heatmap/filter UI rendered on the page. |
| `src/pages/changelog.tsx` | the `/changelog` index page (deliberately SHADOWS the changelog blog-instance's auto-index; the "page already exists at /changelog" build warning is intentional, documented in docusaurus.config.js). |
| Handbook sidebar | a `{type:'link', label:'📓 Changelog', href:'/changelog'}` doorway in `sidebars-handbook.js` (last, after the autogenerated reference docs). |
| `changelog-archive-reminder` Stop hook | counts completed tasks from the transcript and nudges when you cross 10; it does NOT do the work. |

## The batch format (the contract the generator parses)

Append a batch to `CLAUDE-CHANGELOG.md` in EXACTLY this shape (the file's own header documents it):

```markdown
## [YYYY-MM-DD] Short batch title
<!-- meta: type=feature category=development priority=high component=Site -->
One-sentence summary of the batch (becomes the card's description).

- Completed task subject
- Completed task subject
...
```

- The `## ` heading is the split point; the date drives the card's date. Brackets around the date
  are conventional.
- `<!-- meta: ... -->` sets the card's facets. Defaults if omitted: `type=chore`,
  `category=development`, `priority=medium`, `component=Claude`. Common: `type=feature|fix|chore`,
  `component=Site|Claude`, `priority=high|medium|low`.
- The one line right after meta is the **summary** (the card description). The bullets are the
  completed task subjects (lead each with the question/subject the task answered).

## ⚠️ The summaries RENDER ON-SITE, so two rules are non-negotiable

The summary + bullets are compiled and shown to readers on `/changelog`. So:

1. **No raw `<Tag>` (MDX build-breaker).** A literal `<Gif>`, `<Card>`, `<UsedIn>` etc. in a
   summary is parsed as an unclosed JSX tag and **FAILS the prod build** (a real break this repo
   hit: `<Gif>` in a batch summary aborted the build with "Expected a closing tag for `<Gif>`").
   **Wrap any component/tag mention in backticks**: `` `<Gif>` ``. Same for anything that looks
   like a tag or uses `{braces}`.
2. **No literal em-dash `—` (the AI-voice tell).** Reader-facing text with a literal em-dash reads
   as AI-written and is caught by the blocking em-dash hook. Use commas, colons, periods, or
   parentheses. (This is why archive summaries are written de-em-dashed.)

Both are easy to violate when pasting a task subject verbatim. Scan the batch for `<` and `—`
before regenerating.

## The archive LOOP (the CLAUDE.md convention, in detail)

When the task list hits **10+ completed tasks** (the Stop hook nudges; it does not act):

1. **Write the batch.** Append a `## [YYYY-MM-DD] Title` batch to `CLAUDE-CHANGELOG.md`: a meta
   line, a one-line summary, and the completed task subjects as bullets. Convert any relative date
   to absolute. **De-em-dash + backtick-escape** per the rules above.
2. **Regenerate.** `node bytesofpurpose-blog/scripts/generate-changelog-data.js` (or
   `make generate-assets` / any build, which runs it via `prebuild`). This splits the file into
   cards. The output JSON is gitignored, so the change you COMMIT is the CLAUDE-CHANGELOG.md batch,
   not the JSON.
3. **THEN delete the archived tasks** (TaskUpdate → `deleted`) so the live list stays short. The
   archive is only half-done until the move is followed by the delete. Leave pending/in_progress
   tasks untouched.

The completed task subjects ARE the changelog bullets, which is why disciplined task tracking
(the "track our work as tasks" convention) is what makes this step possible. Frame each task
around the question it answered (lead its description with "ANSWERED:") so the bullet reads well.

## Playbook: a changelog card renders wrong / the build breaks on the changelog

| Symptom | Cause | Fix |
|---|---|---|
| Prod build fails "Expected a closing tag for `<X>`" pointing at CLAUDE-CHANGELOG.md | a raw `<X>` tag in a summary/bullet | backtick-escape it (`` `<X>` ``), regenerate |
| em-dash hook blocks the edit | literal `—` in a summary/bullet | rephrase with comma/colon/period/parens |
| A batch didn't become a card | the `## ` heading is malformed (not `## ` at line start) or the file wasn't regenerated | fix the heading; run `generate-changelog-data.js` |
| Card has wrong facets/date | missing or wrong `<!-- meta: -->` / date | fix the meta line + the `[YYYY-MM-DD]` |
| Edit to changelog-data.json blocked | it's GENERATED | edit CLAUDE-CHANGELOG.md (the source) + regenerate; the block-generated-edits hook enforces this |
| "page already exists at /changelog" warning | the `src/pages/changelog.tsx` shadow (intentional) | not a finding, leave it |

## Playbook: surface / move the changelog page

- The page is **its own blog instance at `/changelog`** with the rich `<Changelog>` UI; don't
  duplicate it. To surface it elsewhere, **link to `/changelog`** (the Handbook sidebar already
  does, via a `{type:'link'}` item in `sidebars-handbook.js`).
- A route change to `/changelog` is a normal move: pair it with a `{from,to}` redirect and let the
  `validate-redirects` gate (now also catching chains) pass.

## Guardrails

- **Source vs generated:** edit `CLAUDE-CHANGELOG.md`, never `changelog-data.json` (generated,
  gitignored, hook-blocked). Commit the source; the JSON rebuilds.
- **Prove it:** after a batch, regenerate and confirm the card count went up (the prove-don't-assert
  tenet); if you touched the page, a build is the real check.
- **The loop isn't done until the delete:** archive → regenerate → DELETE the tasks.
- Owning conventions in CLAUDE.md: "archive completed tasks to the changelog" (the trigger + loop)
  and "track our work as tasks" (the feeder). This skill is the detailed mechanics for both.
