# Manage docs instances (add / rename / split a root resource)

The site's docs are several **root instances**, each its own `@docusaurus/plugin-content-docs` with its
own navbar tab and sidebar, served at its own route: **Craft** (`/craft`), **Journey** (`/journey`,
plugin id `self`), **Knowledge** (`/knowledge`), **Habits** (`/habits`), **Handbook** (`/handbook`,
plugin id `legend`). Clicking a navbar tab shows ONLY that instance's tree. `reorganize-content` moves
docs WITHIN/between instances; this skill is the level above it: standing up a whole NEW root, renaming
a route, or splitting a theme out of an existing instance into its own root.

**When to reach for this:** "make X its own root / its own resource", "split X out into /X", "add a new
docs section", "rename the /Y route". If the user just wants a doc moved to an existing home, that's
`reorganize-content`.

## The 7-touchpoint registration checklist (for a NEW instance, `id == routeBasePath`, no prior URL)

Miss one and it won't build or the validators fail. All verified building `/knowledge` + `/habits`.

**Required (won't build otherwise):**
1. **`docusaurus.config.js` plugins array** — a new `@docusaurus/plugin-content-docs` block. Copy an
   existing one (craft's is the template): `id`, `path: 'docs/<name>'`, `routeBasePath: '<name>'`,
   `sidebarPath: require.resolve('./sidebars-<name>.js')`, the shared `remarkPlugins`/`rehypePlugins`
   (`remarkTaskList` / `rehypeTaskListLabels, rehypePremiumEncrypt`), and the repo `editUrl`.
2. **`sidebars-<name>.js`** (new file) — `module.exports = { <name>Sidebar: [{type:'autogenerated', dirName:'.'}] }`.
   The exported key is the `sidebarId` the navbar references.
3. **`docusaurus.config.js` navbar.items** — a `docSidebar` item: `type: 'docSidebar'`,
   `sidebarId: '<name>Sidebar'`, `docsPluginId: '<name>'` (MUST equal the plugin `id`), `label`,
   `position: 'left'`. (Blog instances use a plain `to:` link instead; docs instances use `docSidebar`.)
4. **`docs/<name>/`** — the content dir with a landing `README.mdx` (absolute `slug: /`, pinned as
   Welcome — see below) + `_category_.json`.

**Required for the validators/generators (parallel hand-maintained instance lists — each mirrors the
config; keep them in lockstep):**
5. **`scripts/validate-redirects.js`** `DOCS_INSTANCES` — add `{dir:'docs/<name>', base:'/<name>'}`.
6. **`scripts/validate-seo.js`** `DOCS_ROOTS` — add `'docs/<name>'`.
7. **`scripts/generate-component-usage.js`** `DOCS_INSTANCES` — add `{dir, base}` **AND** confirm the
   base→dir resolution is a LOOKUP, not a ternary. (It used to be
   `base === '/craft' ? … : 'docs/handbook'`, which SILENTLY misroutes any new instance to handbook.
   It's now `DOCS_INSTANCES.find(i => i.base === base).dir` — keep it a lookup.)

**Conditional:**
8. **`scripts/generate-todos-data.js`** `SCAN` — add the instance if its `- [ ]` tasks should board.
9. **`createRedirects` (config) + `validate-redirects.js` `WILDCARD_REWRITES`** — ONLY when RENAMING a
   route (a `/oldprefix/* -> /newprefix/*` backstop). A brand-new root needs NEITHER.
10. **`src/pages/index.tsx` `CHOOSER_CARDS`** — a homepage chooser card (see "Visual identity"). If you
    want it ENFORCED, add the route to `validate-docs-structure.js` `checkWelcomeDrift`.
11. **`test/e2e/*`** — extend `craft-self-split.spec.ts` / `homepage.spec.ts` if you want e2e coverage.

**No change needed (generic, first-path-segment-derived):** `plugins/draft-docs/index.js` (derives the
instance from the first `docs/` segment), `scripts/lib/premium-docs.js`, `scripts/validate-links.js`, and
the topic discovery in `scripts/validate-docs-structure.js`.

## Moving content INTO the new instance (each doc = a reorganize-content move)

Splitting a theme out is a CROSS-INSTANCE `reorganize-content` per doc: `git mv` into `docs/<name>/...`,
rewrite the slug (see the instance-relative gotcha), fix internal links, and **redirect every PUBLISHED
old URL** to its new home (repoint + collapse existing chains — the redirect gate bites). Fold each
source section README, don't drop it; prune the emptied shells (`prune-content`). Regenerate assets +
validate after each batch. See `reorganize-content` for the per-doc mechanics; this skill just adds the
instance around them.

## Visual identity (navbar + the chooser card)

- **Navbar:** pick a distinct **emoji + short label** (`🧠 Knowledge`, `🔄 Habits`). The label carries
  the emoji (unlike docs, a navbar item's emoji is hand-typed in the label).
- **Homepage chooser card (`CHOOSER_CARDS`, deferred by default):** each root can have an **arched card
  image** at `static/img/cards/<name>.png`, shown on the homepage hero chooser (like `craft.png`,
  `self.png`). The art must ABIDE BY the canonical arch (1024², arched silhouette, transparent outside)
  so it drops into the hero like the others — see **`import-arched-image`** (the fit-to-arch script) and
  **`maintain-homepage-hero`** (the `CHOOSER_CARDS` array + the arch mask). To ADD a card: append a
  `{to, img, title, body}` entry to `CHOOSER_CARDS` and drop the fitted PNG in `static/img/cards/`. If
  you have no art yet, LEAVE the chooser at its current cards and reach the new root via the navbar (what
  `/knowledge` + `/habits` did) — the chooser card is a follow-up, not a blocker.

## Gotchas (each cost real time building /knowledge + /habits)

- **The slug MUST be INSTANCE-RELATIVE, not absolute-with-prefix.** A doc in the `/knowledge` instance
  has `slug: /ai/fundamentals` (the base `/knowledge` is prepended by the instance), NOT
  `slug: /knowledge/ai/fundamentals` — an absolute-with-prefix slug DOUBLES to
  `/knowledge/knowledge/ai/...` and every redirect `to:` it becomes `redirect-to-missing`. The landing
  README's slug is `slug: /` (publishes at the instance root). This mirrors the craft convention (a
  craft doc is `slug: /software-development/...`, not `/craft/software-development/...`).
- **A long-running dev server has a STALE ROUTE TABLE — restart it after adding an instance.** `yarn
  start` (`:3000`) caches its route table at startup: a newly-added instance / a moved route / a moved
  draft will NOT show (and the OLD url keeps 200-ing) until you kill + restart the dev server. This is
  the #1 "why don't I see my new pages/drafts" confusion. (The prod build `:4173` is a fresh static
  snapshot each `make build`, but ALSO excludes drafts — so drafts only ever show on a FRESHLY-restarted
  `:3000`.) See `serve-locally`.
- **Drafts don't ship to prod.** Most freshly-split content is `draft: true`; it shows on `:3000` but
  NOT in the prod build. So a mostly-draft new instance looks near-empty on `:4173` — that's correct, not
  a bug. A `draft: true` doc has no public URL, so it needs NO redirect (and the gate REJECTS a redirect
  `to:` a draft — point such a redirect at the nearest PUBLISHED page, e.g. the group landing, until it
  un-drafts).
- **Pin the landing as `👋 Welcome`.** Give the instance's root `README` `sidebar_label: '👋 Welcome'` +
  `sidebar_position: 0` so it sits FIRST in the sidebar (matching Craft/Journey). Without it the landing
  sorts last under its bare title.
- **`docsPluginId` must equal the plugin `id`; `sidebarId` must equal the sidebar file's exported key.**
  A mismatch renders an empty navbar tab. (Note two existing instances keep a legacy `id` ≠ route:
  Journey is `id: 'self'` at `/journey`, Handbook is `id: 'legend'` at `/handbook` — for a NEW instance,
  set `id == routeBasePath` to avoid that indirection.)
- **The em-dash content hook bites landing READMEs.** A landing is reader-facing content, so a literal
  `—` is BLOCKED — use a colon/period in the "What's here" list (see `review-reader-experience`).

## Verify a new instance end-to-end
1. `( cd bytesofpurpose-blog && npm run generate-assets )` then `make build` — the new `/<name>` route
   renders, the navbar shows its tab, "Broken links" is empty.
2. `make validate-redirects` (clean) + `make validate-seo` + `make validate-structure`.
3. **Restart `:3000`** (`make start`), then click the new navbar tab: it shows ONLY its own tree, the
   pinned Welcome is first, a moved doc resolves, and an old URL 301s to the new home.
4. Standing workflow: branch → PR (list the touchpoints + the redirect count + a navbar screenshot) →
   ask to merge → squash.

## Files this skill spans (owned across the config + scripts)
`docusaurus.config.js` (the plugin block + navbar item + redirects), `sidebars-<name>.js`, the parallel
instance lists in `scripts/{validate-redirects,validate-seo,generate-component-usage,generate-todos-data}.js`,
`docs/<name>/**`, and (for the chooser card) `src/pages/index.tsx` + `static/img/cards/`. The two-instance
convention is documented in CLAUDE.md ("structure decisions must update the structure checks"); this skill
is the how-to for changing the instance SET. Pairs with `reorganize-content` (per-doc moves),
`import-arched-image` + `maintain-homepage-hero` (card art), `prune-content` (the shells a split leaves).
